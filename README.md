# URLとは何か
 WEBサイトのページがどこに位置するのかを示す「アドレス」的な意味合いで利用される。
URN（Uniform Resource Name）と合わせてURIと呼ばれる。

 URLは主に http:// や https://(secure) から始まり、そのあとにドメイン名が続く形で構成される。<br>
 URLの構成としてはスキーム・ホスト名（ポート番号）、パス、クエリパラメータ、コンテンツ名を
含み、上記要素を含むことでWEBサイト上の詳細なページの位置を示すことができる。

 WEBサイトの所有者や管理者がURLが示すWEBページやファイルを削除することは可能だが、URLそのものを削除することはできない。ただしURLが検索エンジンの検索結果から消えるようにすることは可能。

 URLには<span style="color: red; ">「絶対URL」「相対URL」</span>がある。<br>
* 絶対URL<br>
「http」「https://」などのスキームから始まる。<br>
* 相対URL<br>
  スキームから始まらないものも含む。/から始まるものなども多い。<br>

# クエリ文字列とは何か <br>
サーバーに情報を送るためにURLの末尾に付け足す文字列のこと。<br>
「？」をURLの末尾に着けその次に「パラメータ=値」をつける。複数のパラメータをつけたい場合は「＆」を使用する。<br>
「？」自体や空白、日本語文字など値の表記に使用できない文字を含めたい場合は<span style ="color:blue; ">「URLエンコード」「パーセントエンコーディング」</span>と呼ばれる方式で安全な文字列に変換する。<br><br>
### パーセントエンコーディング <br>
URIは世界中で使用されるため、どの言語でも使える文字しか利用できないという決まりがある。<br>
そのため、カタカナやひらがな、漢字といった海外環境で使えない可能性がる文字列は含んでいてはならず、それを変換するために利用されている。
表現方法は％と16進数で表現される。` Ex）？→%3F、スペース→%20など `<br>
変換された文字列はサーバー側で元に戻され、元の日本語文字列として使用される。<br>

## ` クエリ文字列には2種類　<br>
## ` 「パッシブパラメータ」「アクティブパラメータ」 `<br>

### 1. パッシブパラメータ<br>
  コンテンツの内容に影響を与えない。ページが変化しないためダミーパラメータとも呼ばれる。<br>
  パッシブパラメータをつけることでより詳細なアクセス解析が可能になる。<br>
  Ex)ユーザーの流入元、メルマガ・スマホアプリなどの流入元の媒体、どの位置の広告をクリックしたかなど<br>
  
### 2. アクティブパラメータ<br>
  パッシブパラメータとは対照的に表示されているページの内容を変化させることができる。<br>
 ` Ex)価格順・人気順などのソート、種類を絞ったフィルタリングなど ` <br>

 クエリ文字列はhttps化しても暗号化されることがないため、ユーザーのパスワードや個人情報など盗まれてはいけない情報をクエリ文字列で送信するのは危険。またサーバーやブラウザによって文字列を利用できる文字数も決まっているため注意が必要。<br>

# パスパラメータとは<br>
URLパラメータの一種としてURLのパスの中に埋め込まれた変数のようなもので特定のリソースを識別するために必要な情報。<br>
* ユーザーページなど個別のリソースにアクセスするとき<br>
* REST APIのエンドポイントでリソースを特定するとき<br>

等で利用されている。<br>
要はURLのパスの一部を変数化して柔軟なルーティングを実現するための仕組み。

# パスパラメータとクエリパラメータの違い<br>
#### 【パスパラメータ】<br>
→特定のリソースを識別する際に必要な情報。URIでドメインの後、？の前に位置する。
 またパスパラメータはURLの一部を利用してリソースを指定する。<br>
#### 【クエリパラメータ】<br>
→特定のリソースを操作して取得する際に必要な情報。URIで、？の後に位置する。
 key=valueの形式で指定する。

# HTTPメソッドとは何か<br>
　HTTPリクエストを用いてWEBサーバーに具体的な要求をする。<br>
 「HEAD」「GET」「POST」「DELETE」「PATCH」「CONNECT」「OPTIONS」「TRACE」「PUT」がある。<br>
### **1.GET**<br>
  WEBサーバから情報を取り出すために使用される。データが変更されないため安全性と冪等性がある。取得するだけなので、データのキャッシュが可能。GETを使用するリクエストはデータの取り込みに限る。<br>
  GETメソッドを使ってデータを送信するときはURLの後ろに送りたいデータをつけて送るため、送ったデータがWEBブラウザの閲覧履歴に残る。検索などでフォームに値を入れて送信した場合、通常クエリストリングでデータが送信される。<br>
` EX）HTML情報の取得、画像の取得、ファイルの取得、検索結果をJSONで取得するなど ` <br>

### **2.POST**<br>
  WEBサーバへ情報を送るために使用する。何かリソースを追加したり更新したりするため冪等性はない。<br>
  POSTメソッドを使ってデータを送信するときはHTTPリクエスト内のメッセージボディ内にデータを格納して送るため、メッセージボディ内にデータが組み込まれ、閲覧履歴には残らない。このメソッドではフォームに値を入れて送信した場合、通常リクエストボディで送信されるためURLには含まれない。<br>
EX)ファイルをアップロードしてクラウドに保存、動画をアップロードするなど<br>

### **3.PUT**<br>
 POSTと似ているが、PUTには冪等性がある。<br>
 PUTは何らかのリソースを特定の状態にするといった操作に利用される。（何度それを実行しても同じ状態になるようにする）PUTではリソースの全体を送信するため全体をその値で更新するということになる。<br>
 
### **4.PATCH**<br>
 PATCHはリソースを修正するために使用される。<br>
 ユーザーはリソースをどのように修正するかを記述して送信し、サーバはそれを見て処理する。 修正であるため冪等であることが多いが、PUTと異なり冪等である必要はない。<br>
 PATCHでは部分的に変更内容を送信するため、PUTと異なりリソース全体ではなく部分的な更新となる。<br>
 
### **5.DELETE**<br>
 特定のリソースを削除するために使用される。<br>
 冪等性があるため、サーバ側で最新のデータを判定して削除する操作にDELETEメソッドを使うべきではない。（実行するたびにリソースが減り、冪等にならないため）クライアント側で最新データを判別し、サーバ側ではデータのIDを受け取って削除するという命令ではDELETEメソッドを使用できる。（特定のIDのデータはいつどのタイミングでも同じデータを指すため)そのため、何度削除命令を出しても同じ状態になるように実装しなければならない。<br>

# HTTPリクエストヘッダーとは<br>
 WEBコンテンツの伝送に使用されるHTTPでクライアントからサーバへの要求であるHTTPリクエストの前半にある制御情報を示した部分。要求内容の詳細を記述する。<br>
 サーバー側ではその情報をもとに適切なレスポンスを生成することができるため、リクエストヘッダーは重要な役割を持つ。<br>
 ヘッダーによって要求するものが変わる。<br>
* クライアントの情報をサーバーに伝える<br>
  →User-AgentやAccept-Languageなど
* リクエストの内容や形式を特定する<br>
  →Content-TypeやAcceptなど
* 認証やセッション管理に関する情報を伝達する<br>
  →CookieやAuthorizationなど
* リクエスト本文の大きさや形式を特定する<br>
  →Content-LengthやContent-Encodingなど
* クライアントが希望するレスポンスの形式を指定する<br>
  →AcceptやAccept-Encodingなど
  
# HTTPステータスコードとは<br>
 特定のHTTPリクエストが正常に完了したかどうかを表す。<br>
* 100番台:情報レスポンス
* 200番台:成功レスポンス
* 300番台:リダイレクトメッセージ
* 400番台:クライアントエラーレスポンス
* 500番台:サーバーエラーレスポンス<br>
### **【200:OK】** <br>
→リクエストが成功し、レスポンスとともに要求に応じた情報が返される。<br>
　ブラウザでページが正しく表示された場合はほとんどこのステータスコードが返っている。<br>
### **【201:CREATED】** <br>
→リクエストが完了し新たに作成されたリソースのURIが返される。<br>
` EX)PUTメソッドでリソースを作成する陸枝洲男を行ったときそのリクエストが完了した場合などに返される。 ` <br>
### **【400:Bad Request】** <br>
→リクエストが不正。定義されていないメソッドの使用などクライアントのリクエストがおかしい場合に返される。<br>
### **【404:Not Found】** <br>
→リソースやページが見つからなかった。またはアクセス権がない・単に要求をこたえられない利用ウをごまかすためにも使われる。<br>
### **【500:Internal Server Error】** <br>
→サーバ内部エラー。サーバ内部にエラーが発生した場合などに返される。<br>
` EX)CGIとして動作させているプログラムに文法エラーがあったり、設定に誤りがあった場合などに返される。 ` <br>

# HTTPレスポンスヘッダーとは <br>
 HTTPでサーバからクライアントへの応答であるHTTPレスポンスの前半にある制御情報を記した領域のこと。HTTPステータスラインに書ききれないレスポンスの情報が書かれている。<br>
 レスポンスヘッダはクライアントが正確な情報を得るために必要。WEBページを表示する際に必要なコンテンツの種類や文字コードを含めることでクライアントは正確なページの表示に必要な情報を得ることができる。<br>
 またレスポンスヘッダにはセキュリティ関連の情報も含まれており、クライアントに対してサイト内のコンテンツが正規のMIMEタイプであることを強制したり、XSS攻撃やクロスサイトリクエストフォージェリなどのセキュリティリスクを低減することができる。<br>

# HTTPレスポンスボディとは <br>
 サーバからの応答メッセージであるHTTPレスポンスの中で、後半部分の「クライアントへ送信するファイルなどの本体であるボディ部」のこと。リクエストに対するサーバー応答内容の本文。<br>

# JSON(Java Script Object Notation)　<br>
 JavaScriptのオブジェクト記法を用いたデータ交換フォーマット。<br>
JSONを間に挟むことで各プログラミング言語間のデータの受け渡しが容易になる。<br>

### `JSONの特徴` <br>
 []や{}などJavaScriptでオブジェクトを作成する際の記法をもとに記述する。<br>
 JavaScriptなどのクライアント言語とPythonなどのサーバサイド言語観のデータやり取りで用いられることが多い。さらにJSONはXMLと比較して人間が見てもわかりやすく、PythonからJavaScriptにJSONデータを送った時にそのままオブジェクトとして扱うといった利用法も可能。<br>
